# Specification: v2.2 CI/CD Reasoning Guardrails

## 1. Overview
The CI/CD Reasoning Guardrails is a CLI utility designed to integrate into automated pipelines (GitHub Actions, GitLab CI, Jenkins). It shifts TraceWhisper from a manual debugging tool to an automated quality gate, ensuring that prompt changes do not introduce reasoning regressions.

## 2. Core Objectives
- **Prevent Regression:** Ensure that a new system prompt doesn't introduce `[Reasoning Loop]` or `[Contradiction]` events on a known "Gold Standard" dataset.
- **Quantify Quality:** Provide a "Reasoning Efficiency Score" that must stay above a certain threshold for a build to pass.
- **Automated Auditing:** Generate a summary report for every PR that allows reviewers to see the "intelligence delta" before merging.

## 3. Functional Requirements

### 3.1 Gold Standard Dataset Management
- The tool shall support a directory of "Gold Standard" inputs and expected reasoning outcomes (or at least a baseline efficiency score).
- Users can define a `tw-guardrails.json` config file specifying which traces to use as baselines.

### 3.2 The Guardrail Check (The "Gate")
The CLI tool `tw guardrail check` will perform the following:
1. **Execution:** Run the agent with the new prompt against the Gold Standard dataset.
2. **Trace Generation:** Generate traces for all runs.
3. **Analysis:**
    - **Hard Fail:** If any trace contains a `[Reasoning Loop]` or `[Contradiction]` that was not present in the baseline.
    - **Soft Fail:** If the average `Efficiency Score` (Steps to Solution / Token Count) drops by more than X% (configurable).
4. **Exit Code:** Return `0` for success, `1` for failure.

### 3.3 CI Report Generation
- For every run, the tool shall generate a Markdown summary (`reasoning-report.md`) containing:
    - **Pass/Fail Status.**
    - **Comparison Table:** Baseline vs. Current (Total Steps, Avg. Tokens, Error Rate).
    - **Regression Alerts:** Specific instances where the agent entered a loop that the baseline avoided.

## 4. Technical Constraints
- **Statelessness:** The CLI must be able to run in a container without requiring a persistent database, relying on local files or a temporary API session.
- **Performance:** The check must run in parallel across the dataset to minimize CI pipeline latency.
- **Integration:** Must output standard JUnit-style or GitHub Action-style annotations for easy visibility in PRs.

## 5. Acceptance Criteria
- [ ] `tw guardrail check` command successfully returns exit code 1 when a reasoning loop is detected.
- [ ] `tw guardrail check` successfully returns exit code 0 when efficiency is maintained.
- [ ] A Markdown report is produced that clearly shows the delta between the baseline and the current run.
- [ ] The tool can be configured via a `.tw-guardrails.json` file.
