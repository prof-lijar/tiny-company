# Product Specification: v2.2 CI/CD Reasoning Guardrails

**Date:** 2026-05-20
**Status:** Draft
**Priority:** P2 (Medium)
**Goal:** Prevent "reasoning regressions" from reaching production by integrating logic audits directly into the CI/CD pipeline.

## 1. Overview
A reasoning regression occurs when a change to a system prompt or a model update causes an agent to introduce new reasoning loops, contradictions, or significant efficiency drops on a task it previously solved optimally. The CI/CD Reasoning Guardrail is a "Logic Gate" that fails the build if the agent's reasoning path deviates negatively from a defined baseline.

## 2. User Flow

### 2.1 Pipeline Integration
1. **Trigger**: A developer pushes a PR with changes to the agent's system prompt or configuration.
2. **Execution**: The CI pipeline (e.g., GitHub Actions) invokes the TraceWhisper Guardrail CLI:
   `tw guardrail check --dataset gold_standards.json --threshold 0.8`
3. **Audit**: The CLI runs the agent across a set of "Gold Standard" tasks and captures the reasoning traces.
4. **Comparison**: The system compares the new traces against the `gold_standards.json` baseline.
5. **Verdict**: 
   - **PASS**: All traces are within the efficiency threshold and contain no new reasoning loops.
   - **FAIL**: One or more traces introduced a `[Reasoning Loop]` or the Efficiency Score dropped below the threshold.

### 2.2 The Regression Report
If the build fails, TraceWhisper generates a `regression-report.md` attached to the PR:
- **Failed Task**: "User Email Retrieval"
- **Regression Type**: `Reasoning Loop Detected`
- **Comparison**: 
    - *Baseline*: 4 steps to goal.
    - *Current*: 12 steps (Loop detected at Step 5).
- **Evidence**: A snippet of the loop from the current trace.
- **Recommendation**: "The new prompt change likely caused the agent to over-verify the API key. Review the 'Verification' section of your prompt."

## 3. Technical Requirements

### 3.1 Guardrail CLI
- **Headless Execution**: Must run in a non-interactive environment.
- **Dataset Loading**: Ability to load a set of inputs and their corresponding Gold Standard baselines.
- **Exit Codes**: 
    - `0`: All checks passed.
    - `1`: Reasoning regression detected.
    - `2`: Execution error (e.g., API timeout).

### 3.2 Regression Detection Logic
The gate fails if any of the following are true:
1. **New Anomaly**: A `[Reasoning Loop]` or `[Contradiction]` is detected where none existed in the Gold Standard.
2. **Efficiency Drop**: $\frac{\text{Current Efficiency Score}}{\text{Baseline Efficiency Score}} < \text{Threshold}$ (default 0.8).
3. **Milestone Miss**: A required milestone defined in the Gold Standard Registry was skipped.

### 3.3 Integration with Gold Standard Registry
The Guardrail tool must be tightly coupled with the `docs/v2.2-gold-standard-registry-spec.md` to ensure it uses the same "Platonic Ideal" paths for comparison.

## 4. Acceptance Criteria
- [ ] CLI command `tw guardrail check` successfully returns exit code 1 when a loop is introduced.
- [ ] The tool can be integrated into a standard GitHub Action workflow.
- [ ] The regression report clearly identifies the task and the nature of the failure.
- [ ] Efficiency calculations are consistent with the `EfficiencyCalculator` used in the IDE.

## 5. Success Metrics
- **Regression Leakage**: Number of reasoning bugs that reach production after the guardrail is implemented.
- **Build Stability**: Percentage of builds that fail due to reasoning regressions vs. technical errors.
