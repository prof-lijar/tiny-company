# Specification: v2.2 Reasoning Template Library

## 1. Purpose
The Reasoning Template Library is the "Knowledge Base" of TraceWhisper. While the 'Fix-It' engine provides dynamic, LLM-generated suggestions, the Template Library provides **verified, high-confidence patterns** and their proven resolutions. 

The goal is to move from *stochastic correction* (hoping the LLM suggests the right fix) to *deterministic correction* (matching a known failure pattern to a known solution).

## 2. Functional Requirements

### 2.1 Pattern Matching Engine
The system must be able to automatically suggest templates based on the current trace analysis.
- **Symptom Detection**: The engine should scan traces for signatures defined in the library (e.g., the same tool called 3x with the same args $\rightarrow$ "The Infinite Loop").
- **Priority**: Template-based fixes should take precedence over dynamic 'Fix-It' suggestions, as they are "proven" solutions.
- **Confidence Score**: Each match should be presented with a confidence level based on how closely the trace matches the symptom signature.

### 2.2 The Library Interface (UX)
Developers should be able to browse the library independently of a specific trace.
- **Search & Filter**: Filter by Category (Loop, Contradiction, Hallucination, Drift, Tool-Blindness).
- **Pattern Detail View**:
    - **The Symptom**: Clear description of what the failure looks like.
    - **The Root Cause**: The "why" behind the failure.
    - **The Proven Fix**: The exact prompt snippet to be added.
    - **Example Trace**: A link to a "Gold Standard" failure trace for comparison.
- **One-Click Application**: A button to "Apply to Current Prompt" which appends the fix to the developer's active system prompt.

### 2.3 The Contribution Loop (Crowdsourcing)
The library should grow organically from the Beta feedback loop.
- **Promotion Workflow**: When a dynamic 'Fix-It' suggestion is marked as "Solved" and "High Impact" by multiple users, it is flagged for "Template Promotion".
- **Curator Review**: A Product/CTO review process to refine the dynamic fix into a generalized template.
- **User Submission**: Allow Pro/Enterprise users to submit their own "Proven Fixes" to the community library.

---

## 3. Data Model (Technical Spec)

The library will be stored in a structured format (SQLite/JSON) with the following schema:

| Field | Type | Description | Example |
| :--- | :--- | :--- | :--- |
| `template_id` | UUID | Unique identifier | `tmpl_001` |
| `name` | String | Human-readable name | "The Infinite Loop" |
| `category` | Enum | Failure category | `LOOP` |
| `symptom_signature` | String/Regex | The pattern used to detect the bug | `tool_call_{id}_{args}.repeat(3+)` |
| `root_cause` | Text | Explanation of the cognitive gap | "Lack of termination condition" |
| `proven_fix` | Text | The prompt snippet to resolve it | "If you call the same tool... stop." |
| `example_trace_id` | UUID | Reference to a real-world failure | `trace_abc_123` |
| `version` | Integer | Version of the fix | `1` |
| `confidence_weight` | Float | How strongly this fix usually works | `0.95` |

---

## 4. Integration with v2.2 Workflow

The Template Library transforms the "Fix-It" workflow:

**Old Workflow (Dynamic):**
`Trace` $\rightarrow$ `LLM Analysis` $\rightarrow$ `Generated Suggestion` $\rightarrow$ `User Test`

**New Workflow (Template-Augmented):**
`Trace` $\rightarrow$ `Template Engine (Regex/Pattern Match)` $\rightarrow$ **If Match Found** $\rightarrow$ `Proven Fix` $\rightarrow$ `User Test`
$\rightarrow$ **If No Match** $\rightarrow$ `LLM Analysis` $\rightarrow$ `Generated Suggestion` $\rightarrow$ `User Test`

## 5. Success Metrics
- **Template Hit Rate**: % of reasoning failures that are matched to a library template.
- **Template Acceptance Rate**: % of template-based fixes accepted vs. dynamic fixes.
- **Library Growth Rate**: Number of new verified templates added per month during Beta.
