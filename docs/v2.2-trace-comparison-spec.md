# Product Specification: v2.2 Comparative Trace Analysis (A/B Testing)

## 1. Overview
Comparative Trace Analysis allows developers to quantitatively and qualitatively compare two different execution traces of the same agent task. This is critical for prompt engineering, as it allows a developer to prove that a change in the system prompt actually improved the reasoning efficiency and accuracy, rather than just changing the final output by chance.

## 2. User Flow

### 2.1 Triggering a Comparison
1. **Selection**: The user selects two traces from their trace history (e.g., Trace A: Original Prompt, Trace B: Modified Prompt).
2. **Action**: The user clicks **"Compare Traces"** or runs `tw compare <trace_id_a> <trace_id_b>`.
3. **Processing**: The system aligns the two traces based on the input goal and identifies the point of divergence.
4. **Visualization**: The system presents a side-by-side view of the two narratives and a "Comparative Insights" dashboard.

### 2.2 Analysis Process
- **Alignment**: The system maps steps in Trace A to corresponding steps in Trace B.
- **Divergence Point**: The system highlights the exact step where the agent's reasoning path diverged.
- **Metric Calculation**: The system calculates efficiency scores and waste for both traces.

## 3. Input/Output

### 3.1 Input
- **Trace A**: Full JSON log and narrative.
- **Trace B**: Full JSON log and narrative.
- **Baseline Metrics**: (Optional) The "Gold Standard" path if one exists for this specific task.

### 3.2 Expected Output (The Comparison Report)
The output must include:
1. **Divergence Analysis**: 
    - "Traces were identical until Step X. At Step X, Agent A chose Tool Y, while Agent B chose Tool Z."
2. **Quantitative Comparison Table**:
    | Metric | Trace A | Trace B | Delta |
    | :--- | :--- | :--- | :--- |
    | Total Steps | 24 | 18 | -25% $\downarrow$ |
    | Token Usage | 12k | 9k | -25% $\downarrow$ |
    | Time to Goal | 45s | 32s | -28% $\downarrow$ |
    | Reasoning Loops | 2 | 0 | -100% $\downarrow$ |
    | Efficiency Score | 65% | 88% | +23% $\uparrow$ |
3. **Qualitative Verdict**:
    - A summary statement: *"Trace B is significantly more efficient. It avoided the reasoning loop at Step 12 found in Trace A by directly accessing the API instead of searching the documentation first."*
4. **Outcome Comparison**:
    - Did both reach the correct answer? (Success / Success) or (Success / Failure).

## 4. Acceptance Criteria

A Trace Comparison is considered successful if:
1. **Accurate Divergence Detection**: The system correctly identifies the first point where the agents' actions or thoughts differ.
2. **Metric Integrity**: The calculated metrics (steps, tokens, time) match the raw log data exactly.
3. **Actionable Insight**: The report provides a clear "Winner" and a reasoning-based explanation for why that trace won.
4. **Side-by-Side Clarity**: The UI allows the user to scroll through both narratives in sync.

**Success Metric:**
- **A/B Test Frequency**: $> 40\%$ of users utilizing Comparative Analysis for prompt tuning.

## 5. Integration

### 5.1 UI Integration (Dashboard)
- **Side-by-Side Viewer**: A dual-pane narrative view where synchronized scrolling is enabled.
- **Divergence Marker**: A clear visual indicator (e.g., a red vertical line) at the point where the paths diverge.
- **Metric Cards**: Top-level cards showing the Delta ($\Delta$) for key KPIs.

### 5.2 CLI Integration
- **Command**: `tw compare <id1> <id2>`.
- **Output**: A formatted Markdown table of metrics followed by the Divergence Analysis.

### 5.3 Technical Integration
- **Alignment Engine**: A logic layer that can normalize steps across different traces to find logical equivalents.
- **Efficiency Calculator**: A module that computes the "Efficiency Score" based on (Steps to Goal / Minimum Possible Steps).
