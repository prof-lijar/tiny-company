# v2.3 Beta Success Metrics & Feedback Loop

## 1. Primary KPIs (Key Performance Indicators)

The v2.3 Beta focuses on the transition from "fixing" to "optimizing." We will measure success across four dimensions: Intelligence, Efficiency, Stability, and Workflow.

### 1.1 Intelligence (The Pattern Vault)
*   **Pattern Adoption Rate:** % of successful corrections that were initiated by a "Similar Fix" suggestion from the Pattern Vault.
    *   *Target:* $\ge 30\%$ of corrections.
*   **Vault Precision:** % of Vault suggestions that the user actually accepts without significant modification.
    *   *Target:* $\ge 60\%$ acceptance rate.

### 1.2 Efficiency (Cognitive Pruning)
*   **Token Reduction Ratio:** Average reduction in tokens per successful task after applying Pruning Engine suggestions.
    *   *Target:* $\downarrow 20\%$ average token usage.
*   **Accuracy Preservation:** % of pruned traces that maintain the same output quality as the original (measured via the Gold Standard Registry).
    *   *Target:* $100\%$ (Zero regression in quality).

### 1.3 Stability (CRI & Adversarial Synthesis)
*   **CRI Catch Rate:** % of reasoning regressions detected by `tw verify-all` in CI before they reach the Beta user's manual testing.
    *   *Target:* $\ge 80\%$ of regressions caught by CRI.
*   **Adversarial Discovery Rate:** Number of "critical" reasoning edge cases discovered by the Adversarial Generator vs. those discovered by users.
    *   *Target:* 2:1 ratio (AI finds 2 bugs for every 1 user finds).

### 1.4 Workflow (IDE Integration)
*   **Time to Resolution (TTR):** Mean time from "Trace Failure Detected" to "Prompt Fix Committed" using the VS Code Extension.
    *   *Target:* $\downarrow 40\%$ compared to the v2.2 Web UI workflow.
*   **Extension Stickiness:** % of Beta sessions spent in the VS Code Extension vs. the Web UI.
    *   *Target:* $\ge 70\%$ of time spent in IDE.

## 2. Feedback Loop Design

To ensure rapid iteration, we will implement a multi-channel feedback loop.

### 2.1 Collection Channels
1.  **In-App "Signal" Button:** A one-click button in the VS Code Extension and Web UI to mark a specific trace or suggestion as "Helpful" or "Misleading."
2.  **Structured Bug Reports:** A dedicated Beta feedback form (integrated into the IDE) that automatically attaches the failing trace and the current prompt version.
3.  **Weekly Beta Pulse:** A short, 3-question survey sent every Friday to the Beta cohort focusing on "Perceived Value" and "Friction Points."
4.  **Direct Channel:** A dedicated Slack/Discord channel for the Beta cohort for real-time communication.

### 2.2 Routing & Processing
*   **Triage:** All feedback is routed to the **Beta Coordinator** (Product Manager).
*   **Classification:**
    *   *UI/UX Friction* $\rightarrow$ Designer
    *   *Functional Bug/Stability* $\rightarrow$ CTO
    *   *Intelligence Quality/Vault Gaps* $\rightarrow$ Product $\rightarrow$ CTO
*   **Feedback Cycle:**
    *   **Daily:** Triage and critical bug assignment.
    *   **Weekly:** Synthesis of "Pulse" surveys into a priority list for the following week's sprint.

## 3. Success Thresholds for General Availability (GA)

v2.3 will move from Beta to GA once the following "Hard Gates" are passed:

| Gate | Criteria | Threshold |
| :--- | :--- | :--- |
| **Metric Gate** | KPI Achievement | At least 3 of 4 primary KPI targets (Intelligence, Efficiency, Stability, Workflow) are met. |
| **Stability Gate** | Bug Count | Zero P0 (Critical) or P1 (High) bugs open for 7 consecutive days. |
| **Adoption Gate** | User Sentiment | $\ge 75\%$ of Beta users report that the "Intelligence Layer" significantly reduces their manual prompt engineering effort. |
| **Documentation Gate**| User Readiness | v2.3 User Guide and CRI Integration Guide verified as "Clear" by $\ge 5$ Beta users. |
| **Performance Gate** | Latency Impact | The overhead added by the Pattern Vault query must be $< 500\text{ms}$ per trace analysis. |
