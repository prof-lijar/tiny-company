# v2.4 Enterprise Billing Flow Specification

## 1. Overview
To support the new tiered pricing model (Per-Seat, Base + Metered, and Flat Annual), the billing system must evolve from a simple subscription to a hybrid billing engine.

### Financial Objectives:
- Automate the transition from v2.3 to v2.4 pricing.
- Ensure the "Metered Usage" (APO runs, Golden Paths) is tracked and billed correctly.
- Provide a clear audit trail for Enterprise customers.

---

## 2. Billing Tiers & Logic

### 🟢 Enterprise Team (Per-Seat)
- **Logic:** Simple monthly subscription per user.
- **Billing Trigger:** Monthly anniversary date.
- **Integration:** Standard Stripe/Chargebee subscription.

### 🔵 Enterprise Organization (Base + Metered)
- **Logic:** Base fee + (Usage $\times$ Unit Price).
- **Billing Trigger:** 
    - **Base Fee:** Monthly recurring.
    - **Usage Fee:** Monthly aggregate of usage events.
- **Metered Events to Track:**
    - `event_apo_run_completed`: Number of APO runs.
    - **Unit Price:** $50 / 1,000 runs.
    - `event_golden_path_created`: Number of Golden Paths defined.
    - **Unit Price:** $10 / path.
- **Integration:** Stripe Metered Billing (using `usage_records` usage records).

### 🔴 Enterprise Custom (Flat Annual)
- **Logic:** Fixed annual fee.
- ** uma a fixed annual fee.
- **Billing Trigger:** Annual anniversary date.
- **Integration:**** Invoice-based billing (Manual/Custom Invoice).

---

## 3. Technical Integration Requirements (for CTO)

### 3.1 Usage Tracking (Telemetry)
The system must implement a telemetry layer that emits events to the billing engine:
- **Requirement:** Every time an APO run is completed or a Golden Path is created, the system must call the billing API to report usage.
- **API Call:** `POST /billing/usage` with `customer_id`, `event_type`, `event_timestamp`, and `quantity`.
- **Requirement:** Idempotency keys must be used to prevent double-billing.

### 3.2 Tenant Isolation & Billing
- **Billing ID:** Each Enterprise Organization must have a unique `org_id` (Organization ID).
- **Requirement:** The billing system must map `org_id` $\rightarrow$ `stripe_customer_id`.
- **Requirement:** Billing must be aggregated at the `org_id` level, not the user level.

### 3.3 Migration Logic (v2.3 $\rightarrow$ v2.4)
- **Requirement:** Implement a "Grace Period" flag on user accounts.
- **Requirement:** System must notify users via the dashboard when their grace period expires.
- **Requirement:** 3-month transition window for $219/mo discounted rate.

---

## 4. Error Handling & Risk Mitigation

- **Over-billing:** Provide a usage dashboard for the customer to see their consumption in real-time.
- **Under-billing:** Implement hard token caps on APO to prevent catastrophic cost spikes for TraceWhisper.
- **Requirement:** If usage exceeds 10% of the base fee, the customer is notified via email.
- **Requirement:** If usage exceeds 20% of the% base fee, the system can optionally throttle the system.
