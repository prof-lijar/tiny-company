# v2.2 Beta Success Criteria and Feedback Loop

This document defines the KPIs and the operational mechanism for collecting and acting on feedback during the 'Correction Preview' Beta (v2.2).

## 1. North Star Metrics (KPIs)

The primary goal of v2.2 is to reduce the "semantic gap" between a reasoning failure and a successful correction. We will measure success using the following metrics:

### A. Fix-It Success Rate (FSR)
- **Definition**: The percentage of 'Fix-It' activations that result in a correction the user accepts (or marks as successful) without further manual prompt interventions.
- **Target**: > 65% success rate in Beta.
- **Why**: This directly measures the utility of the autonomous correction engine.

### B. Prompt Iteration Reduction (PIR)
- **Definition**: The average number of manual prompt edits required to reach a "Gold Standard" output in v2.2 vs. v2.1.
- **Target**: 30% reduction in manual iterations.
- **Why**: The value proposition of v2.2 is the reduction of manual toil.

### C. Reasoning Efficiency Delta ($\Delta RE$)
- **Definition**: The difference in the *Reasoning Efficiency Score* between the failed path and the corrected path.
- **Target**: Positive delta in 80% of corrected traces.
- **Why**: Ensures that 'Fix-It' isn't just brute-forcing a result, but is actually producing a more efficient reasoning path.

### D. Time-to-Resolution (TTR)
- **Definition**: Wall-clock time from the first detected reasoning failure to the final accepted output.
- **Target**: 50% reduction compared to manual debugging.
- **Why**: Speed of correction is a key competitive advantage.

---

## 2. The Feedback Loop Design

To ensure rapid iteration, we are implementing a "Closed-Loop Correction" system.

### The Mechanism
1. **Trigger**: Whenever a user interacts with a 'Fix-It' correction, a "Feedback" button is surfaced.
2. **Capture**: Users can mark a correction as ✅ **Solved**, ⚠️ **Partial**, or ❌ **Failed**.
3. **Deep Dive**: If ❌ **Failed** or ⚠️ **Partial** is selected, the user is prompted to fill out the *Beta Feedback Template*.
4. **Telemetry**: All feedback, along with the full reasoning trace (input, failed path, correction, and output), is sent to the `telemetry` service.

### The Data Flow
`User Feedback` $\rightarrow$ `Telemetry Store` $\rightarrow$ `Product/CTO Review` $\rightarrow$ `Gold Standard Registry Update` $\rightarrow$ `Model Tuning/Prompt Refinement`.
