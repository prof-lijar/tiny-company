# v2.2.1 Quick Win Technical Specification

**Date:** 2026-05-20
**Status:** Draft / For Implementation
**Priority:** P0-Critical
**Goal:** Maximize the "Aha! Moment" for Beta Cohort 1 by replacing stochastic LLM fixes with deterministic, proven corrections for common failure patterns.

## 1. Overview
v2.2.1 is a surgical release designed to harden the 'Fix-It' loop. Instead of relying solely on a dynamic LLM to suggest a fix for a failed trace, the system will first check if the failure matches a known "Seed Pattern." If a match is found, the proven fix is delivered with high confidence.

## 2. P0: Deterministic Correction Engine (Critical)

### 2.1 Functional Requirements
The `CorrectionEngine` must be updated to implement a **Priority Override** mechanism.

**Logic Flow:**
1. **Trace Analysis**: When a trace is flagged for correction, the engine scans the trace for the 5 Seed Patterns.
2. **Pattern Matching**:
    - **Infinite Loop**: Detect 3+ identical tool calls with same arguments.
    - **Direct Contradiction**: Detect opposing assertions within 3 steps.
    - **Tool-Output Blindness**: Detect a tool returning a clear error/value followed by a 'Thought' that ignores it.
    - **Goal Drift**: Detect >3 steps of activity unrelated to the primary goal.
    - **Hallucination Spiral**: Detect a call to a tool not present in the session's toolset.
3. **Override Logic**: 
    - If a match is found $\rightarrow$ Retrieve the **Proven Fix** from the `Reasoning Template Library`.
    - Set `confidence_score = 1.0` (Deterministic).
    - Present the Proven Fix as the primary recommendation, bypassing the dynamic LLM generation.
    - If no match is found $\rightarrow$ Fall back to dynamic LLM fix generation.

### 2.2 Seed Patterns Data Table
| Pattern | Detection Heuristic | Proven Fix Snippet |
| :--- | :--- | :--- |
| **Infinite Loop** | $\ge 3$ identical tool calls | "If you have called the same tool with the same arguments more than twice... stop and analyze why the result is not changing..." |
| **Direct Contradiction** | Opposing facts in trace | "Before committing to a conclusion, explicitly cross-reference it with your previous thoughts..." |
| **Tool-Output Blindness** | Error returned $\rightarrow$ ignored in Thought | "When a tool returns a result, you must explicitly summarize the key finding from that output in your next 'Thought' block..." |
| **Goal Drift** | $>3$ steps of distraction | "At every 3rd step, explicitly state your current progress relative to the primary goal..." |
| **Hallucination Spiral** | Tool call $\notin$ provided toolset | "You are strictly limited to the tools provided in the current session. Before calling a tool, verify its name and parameters..." |

### 2.3 Success Criteria
- **Template Hit Rate**: $> 20\%$ of triggered 'Fix-It' operations are resolved via Seed Patterns.
- **FAR (Fix Acceptance Rate)**: Increase in the percentage of fixes applied by users.

## 3. P1: Telemetry & Value Validation (High)

### 3.1 Functional Requirements
Implement high-fidelity tracking for the "Aha! Moment."

- **`ev_correction_success` Event**:
    - **Trigger**: Triggered when a user applies a fix (via 'Fix-It' $\rightarrow$ 'Apply') AND the subsequent trace reaches a `SUCCESS` state.
    - **Payload**: `{ "pattern_id": "...", "time_to_success": "...", "is_deterministic": true/false }`
- **Micro-Feedback Trigger**:
    - **Trigger**: Immediately after a successful fix application.
    - **UI**: A simple binary prompt: *"Did this fix save you time? [Yes] [No]"*
    - **Event**: `ev_user_value_signal`

### 3.2 Success Criteria
- **Value Proof**: $\ge 30\%$ of Beta users experience at least one `ev_correction_success` event.

---

## 4. P2: UX Polish (Medium)

### 4.1 Functional Requirements
- **One-Click Apply**: The "Apply to Prompt" button must automatically inject the fix into the system prompt without requiring the user to copy-paste.
- **Pattern Callouts**: When a deterministic fix is presented, the UI must label it clearly:
    - **Label**: `Known Pattern Detected: [Pattern Name]`
    - **Subtext**: *"This is a proven fix for a common reasoning trap."*

### 4.2 Success Criteria
- **Time-to-Fix**: Reduction in the average time between "Failure Detected" and "Fix Applied."

## 5. Technical Implementation Path
1. **Data**: Populate `src/template_library.py` (or the SQLite db) with the 5 Seed Patterns.
2. **Engine**: Modify `src/correction_engine.py` to implement the pattern matching and override logic.
3. **Telemetry**: Update `src/telemetry.py` to include `ev_correction_success` and `ev_user_value_signal`.
4. **UI/CLI**: Update the output formatting to include "Known Pattern" callouts.
