# Success Metrics & Acceptance Criteria: v2.2.2 Verification Loop

**Status:** Draft
**Priority:** P0
**Owner:** [Product]
**Reference:** `docs/v2.2.2-verification-loop-spec.md`

## 1. Overview
The v2.2.2 Verification Loop transforms the "Fix-It" experience from a manual trial-and-error process into an automated validation pipeline. Success is defined by the reduction of friction in the prompt-tuning cycle and the elimination of "whack-a-mole" regressions.

## 2. Key Performance Indicators (KPIs)

### 2.1 Efficiency Metrics
*   **Time-to-Resolution (TTR):** The average time elapsed from the moment a reasoning failure is detected to the moment a "Fixed & Stable" verdict is reached. 
    *   *Target:* >50% reduction compared to the manual v2.2 "Fix-It" workflow.
*   **Iteration Count:** The average number of prompt changes required to resolve a failure.
    *   *Target:* Reduction in the number of manual "Apply $\rightarrow$ Test $\rightarrow$ Tweak" cycles.

### 2.2 Quality Metrics
*   **First-Pass Accuracy (FPA):** The percentage of "Apply & Verify" clicks that result in a "Fixed & Stable" (Green) verdict on the first attempt.
    *   *Target:* >60% (acknowledging the inherent volatility of prompt engineering).
*   **Regression Detection Rate:** The percentage of regressions (failures in the Verification Set) that are correctly identified by the loop before the user accepts the prompt.
    *   *Target:* 100%. No regression should be marked as "Success" if it exists in the Verification Set.
*   **Rollback Rate:** The percentage of verification cycles that end in a `ev_verify_rollback` event.
    *   *Target:* Monitoring metric; high rates indicate poor "Fix-It" suggestion quality.

## 3. Acceptance Criteria (Definition of 'Shipped')

### 3.1 Functional Requirements (Hard Gates)
The feature is considered "Shipped" only when the following are verified:
- [ ] **Prompt Control:** The SDK successfully updates the system prompt and reverts to the `last_known_good_prompt.bak` without requiring a manual restart of the agent.
- [ ] **Automated Execution:** The system can trigger a re-run of the specific triggering input and the entire "Verification Set" without user intervention.
- [ ] **Quantitative Verdict:** The "Verdict" (Green/Yellow/Red) is derived from a programmatic comparison of traces (e.g., `Loop` $\rightarrow$ `No Loop`) and Gold Standard matching.
- [ ] **State Management:** The UI correctly transitions through the "Verifying..." state and provides a clear, single-click "Rollback" path.
- [ ] **Telemetry Integrity:** All four core events (`ev_verify_start`, `ev_verify_success`, `ev_verify_fail`, `ev_verify_rollback`) are firing with correct metadata.

### 3.2 Performance & Reliability Benchmarks
- [ ] **Overhead:** The orchestration logic (excluding LLM inference time) adds $< 2$ seconds of latency to the total verification cycle.
- [ ] **Atomic Rollbacks:** Rollbacks must be atomic; there should be no state where the system is left with a "partial" or corrupted prompt.
- [ ] **Zero Zombie States:** The system must handle LLM timeouts or crashes during verification by transitioning to a "Failed" state rather than hanging in "Verifying...".

### 3.3 UX Benchmarks
- [ ] **Click-to-Fix Path:** A user should be able to move from "Suggestion" to "Verified Fix" in exactly 2 clicks ("Apply & Verify" $\rightarrow$ "Accept Fix").
- [ ] **Verdict Clarity:** Internal Beta testers must report 100% clarity on the meaning of the Red/Yellow/Green verdicts without referring to documentation.

## 4. Validation Plan
1.  **Synthetic Regression Test:** Create a prompt that fixes a loop in Trace A but breaks a known success in Trace B. Ensure the loop returns a "Yellow (Regression)" verdict.
2.  **Recovery Test:** Trigger a failure, apply a bad fix, and execute a rollback. Verify that the prompt is identical to the pre-fix state.
3.  **Latency Audit:** Measure the time from "Apply & Verify" click to "Verdict" appearance across 10 different prompts.
