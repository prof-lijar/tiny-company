# v2.2 Beta Validation Plan: The Correction Preview

## 1. Objective
The goal of the v2.2 "Correction Preview" Beta is to validate the core value proposition of the **Fix-It** button: *Can TraceWhisper provide automated prompt suggestions that are actionable, accurate, and reduce the iteration time for prompt engineers?*

We are moving from "Detection" (v2.1) to "Correction" (v2.2). This plan defines how we measure success and validate the Meta-Prompt engine before a full v2.2 release.

## 2. Target Validation Cohort
We will invite a "Correction Alpha" group consisting of:
- **10-15 Power Users** from the v2 Beta cohort.
- **Criteria**: Users who have generated > 50 narratives and have explicitly reported "difficulty in knowing how to fix a loop" in their feedback.
- **Goal**: High-signal feedback from users who are already familiar with the narrative format.

## 3. The Validation Dataset ("The Gauntlet")
To avoid "vibes-based" validation, we will use a fixed set of **Known Reasoning Failures**. The Meta-Prompt engine must be tested against these specific patterns:

| Pattern ID | Failure Type | Description | Expected Fix Type |
| :--- | :--- | :--- | :--- |
| **RF-01** | Infinite Loop | Agent calls the same tool with same args 3+ times. | Termination condition / Exit criteria |
| **RF-02** | Direct Contradiction | Agent asserts X, then asserts NOT X in the next step. | Verification step / Cross-check instruction |
| **RF-03** | Tool-Output Loop | Agent ignores tool output and repeats the same tool call. | Instruction to parse specific field in output |
| **RF-04** | Goal Drift | Agent starts solving A, but shifts to solving B. | Focus/Constraint reinforcement |
| **RF-05** | Hallucination Loop | Agent invents a fact, then tries to "verify" it using a tool that doesn't exist. | Grounding instruction / Tool-set validation |

## 4. Measurement Framework

### 4.1 Primary Metric: Fix-It Acceptance Rate (FAR)
**Definition**: The percentage of suggested fixes that the user accepts as "Correct and Actionable" without requiring significant manual rewriting.

$$\text{FAR} = \frac{\text{Accepted Fixes}}{\text{Total Suggestions Generated}} \times 100\%$$

**Target**: $> 25\%$ (As defined in `docs/v2.2-roadmap.md`).

### 4.2 Secondary Metrics
- **Time-to-Fix (TTF)**: Time elapsed from detecting the loop to applying the fix. (Target: $\ge 50\%$ reduction vs manual brainstorming).
- **Regression Rate**: % of suggested fixes that solve the current loop but introduce a new reasoning error. (Target: $< 10\%$).
- **Actionability Score**: A 1-5 scale where users rate: *"How easy was it to apply this suggestion to my prompt?"* (Target: Average $> 3.5$).

## 5. Feedback Loop & Iteration Process
The validation will follow a weekly sprint cycle:

1. **Deployment**: Release the "Fix-It" button to the Alpha group.
2. **Collection**: Users flag suggestions as `[✅ Fixed]` or `[❌ Unhelpful]`.
3. **Analysis**: For every `[❌ Unhelpful]` result, the Product and CTO agents will:
    - Analyze the trace and the failing suggestion.
    - Identify the "Reasoning Gap" in the Meta-Prompt.
4. **Tuning**: Update the `META_PROMPT` in the `CorrectionEngine` to address the gap.
5. **Re-test**: Run the updated prompt against the "Gauntlet" dataset.

## 6. Exit Criteria for Beta
The "Correction Preview" is considered successful and ready for the **Full v2.2 Reasoning IDE Release** when:
- [ ] **FAR** is consistently $> 25\%$ across the "Gauntlet" dataset.
- [ ] **Regression Rate** is $< 10\%$.
- [ ] **User Sentiment**: $\ge 70\%$ of Alpha users agree that the "Fix-It" button is a "Must-Have" feature.
- [ ] **Technical Stability**: The transition from `_simulate_parsing` to real Pydantic parsing is verified and stable.
