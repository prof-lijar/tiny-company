# Technical Specification: The Verification Loop (v2.2.2)

**Status:** Revised (Integrating Beta Synthesis)
**Priority:** P0
**Owner:** [Product]

## 1. Overview
The "Verification Loop" evolves the v2.2 'Fix-It' feature from a passive suggestion into an active validation workflow. Instead of the user manually applying a fix and re-running their agent to see if it worked, TraceWhisper will orchestrate the application and verification process.

**Goal:** Reduce the Time-to-Resolution (TTR) and eliminate "Whack-a-Mole" regressions by automating the "Snapshot $\rightarrow$ Try $\rightarrow$ Verify $\rightarrow$ (Optional) Rollback" cycle.

## 2. The Verification Workflow
The loop consists of six distinct stages:

1.  **Trigger**: A reasoning failure (e.g., Infinite Loop) is detected in a trace.
2.  **Suggestion**: The system proposes a 'Fix-It' prompt adjustment.
3.  **Approval**: The user clicks "Apply & Verify".
4.  **Safe Execution**: 
    - **Snapshot**: The system saves the current system prompt to a temporary backup.
    - **Push**: TraceWhisper pushes the prompt update to the agent's configuration.
    - **Re-run**: TraceWhisper triggers a re-run of the triggering input.
    - **Regression Check**: TraceWhisper triggers re-runs of the user's "Verification Set" (mini-benchmark).
5.  **Comparison**: The system compares the *Original Trace* and *Benchmark Traces* with the *New Traces*.
6.  **Verdict & State**: 
    - If successful: Prompt is kept; snapshot is archived.
    - If failed/regressed: User is notified and given a one-click "Rollback" option to restore the snapshot.

## 3. Technical Requirements

### 3.1 Integration Layer (The "Push & Pull" Mechanism)
To enable safe updates, TraceWhisper needs bidirectional control over the system prompt.
- **Requirement**: Implement `prompt_update` and `prompt_rollback` hooks in the SDK.
- **Action**: The SDK must support dynamic reload of the system prompt and maintain a local `last_known_good_prompt.bak` file.

### 3.2 The Verification Engine
The system must programmatically determine if a fix was successful across multiple dimensions.

- **Success Criteria (Local)**:
    - **Loop Resolution**: If the original trace had a `[Reasoning Loop]`, the new trace must not have one.
    - **Step Reduction**: Total steps $\le$ original trace (unless the original was a loop).
    - **Outcome Match**: Final output matches the "Gold Standard" (the expected correct answer for that specific input).

- **Success Criteria (Global/Regression)**:
    - **Benchmark Stability**: All traces in the user's "Verification Set" must maintain their "Success" status and not introduce new loops or contradictions.

- **Failure Criteria**:
    - The new trace still contains a `[Reasoning Loop]`.
    - The agent reaches a `[Contradiction]` that wasn't there before.
    - Any trace in the Verification Set fails.

### 3.3 Telemetry Events
- `ev_verify_start`: Triggered when "Apply & Verify" is clicked.
- `ev_verify_success`: Triggered when the new trace resolves the failure AND passes benchmarks.
- `ev_verify_fail`: Triggered when the new trace fails or introduces a regression.
- `ev_verify_rollback`: Triggered when the user reverts to the snapshot.

## 4. UI/UX Specifications

### 4.1 The "Verification State"
While the loop is running, the UI transitions to a "Verifying..." state.
- **Visuals**: Loading indicator with text: *"Applying fix and running regression benchmarks..."*
- **Real-time Feedback**: A checklist showing:
    - [ ] Triggering Trace (Processing...)
    - [ ] Benchmark 1 (Pending)
    - [ ] Benchmark 2 (Pending)

### 4.2 The Comparison View (The "Verdict")
- **Green (Success)**: "✅ **Fixed & Stable!** The loop is resolved and no regressions were found."
- **Red (Failure)**: "❌ **Unresolved.** The fix did not eliminate the loop. [Try another suggestion]"
- **Yellow (Regression)**: "⚠️ **Fixed but Unstable.** The loop is gone, but Benchmark X now fails. [Rollback to Safe State]"

## 5. Acceptance Criteria
- [ ] SDK allows updating and reverting system prompts via API call.
- [ ] System can trigger a re-run of a specific input automatically.
- [ ] System supports a "Verification Set" of multiple inputs for regression testing.
- [ ] The "Verdict" is based on quantitative comparison (Loops $\rightarrow$ No Loops) and Gold Standard matching.
- [ ] UI clearly indicates the "Verifying" state and provides a clear "Rollback" path.
- [ ] Telemetry captures `ev_verify_success`, `ev_verify_fail`, and `ev_verify_rollback`.
