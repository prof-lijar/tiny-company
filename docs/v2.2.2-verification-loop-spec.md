# Technical Specification: The Verification Loop (v2.2.2)

**Status:** Draft
**Priority:** P0
**Owner:** [Product]

## 1. Overview
The "Verification Loop" evolves the v2.2 'Fix-It' feature from a passive suggestion into an active validation workflow. Instead of the user manually applying a fix and re-running their agent to see if it worked, TraceWhisper will orchestrate the application and verification process.

**Goal:** Reduce the Time-to-Resolution (TTR) by automating the "Try $\rightarrow$ Verify" cycle.

## 2. The Verification Workflow
The loop consists of five distinct stages:

1.  **Trigger**: A reasoning failure (e.g., Infinite Loop) is detected in a trace.
2.  **Suggestion**: The system proposes a 'Fix-It' prompt adjustment (via the Seed Pattern Engine or LLM).
3.  **Approval**: The user clicks "Apply & Verify".
4.  **Execution**: 
    - TraceWhisper pushes the prompt update to the agent's configuration.
    - TraceWhisper triggers a re-run of the same input that caused the failure.
5.  **Comparison**: The system compares the *Original Trace* with the *New Trace* using the "Reasoning Efficiency" metrics.

## 3. Technical Requirements

### 3.1 Integration Layer (The "Push" Mechanism)
To enable "Apply & Verify," TraceWhisper needs a way to update the system prompt in the user's environment.
- **Requirement**: Implement a `prompt_update` hook in the SDK.
- **Action**: The SDK must support a dynamic reload of the system prompt without requiring a full application restart.

### 3.2 The Verification Engine
The system must programmatically determine if a fix was successful.
- **Success Criteria**:
    - **Loop Resolution**: If the original trace had a `[Reasoning Loop]`, the new trace must not have one.
    - **Step Reduction**: The total number of steps to reach the goal should be $\le$ the original trace (unless the original was a loop).
    - **Outcome Match**: The final output of the agent must still be correct (validated against the same gold standard or user-defined output).
- **Failure Criteria**:
    - The new trace still contains a `[Reasoning Loop]`.
    - The agent reaches a `[Contradiction]` that wasn't there before.
    - The agent fails to reach the goal entirely.

### 3.3 Telemetry Events
To track the efficacy of the Verification Loop, the following events must be emitted:
- `ev_verify_start`: Triggered when "Apply & Verify" is clicked.
- `ev_verify_success`: Triggered when the new trace resolves the failure.
- `ev_verify_fail`: Triggered when the new trace still fails or introduces a new bug.

## 4. UI/UX Specifications

### 4.1 The "Verification State"
While the loop is running, the UI should transition from the "Suggestion" view to a "Verifying..." state.
- **Visuals**: A loading indicator on the 'Fix-It' button with text: *"Applying fix and re-running trace..."*
- **Real-time Feedback**: A small progress bar showing the new trace being generated.

### 4.2 The Comparison View (The "Verdict")
Once verification is complete, the user is presented with a "Verdict Card":
- **Green (Success)**: "✅ **Fixed!** The reasoning loop was resolved. Steps reduced by X%."
- **Red (Failure)**: "❌ **Unresolved.** The fix did not eliminate the loop. [Try another suggestion]"
- **Yellow (Mixed)**: "⚠️ **Partial Success.** The loop is gone, but the agent took more steps to reach the answer."

## 5. Acceptance Criteria
- [ ] SDK allows updating system prompts via API call.
- [ ] System can trigger a re-run of a specific input automatically.
- [ ] The "Verdict" is based on quantitative comparison of traces (Loops $\rightarrow$ No Loops).
- [ ] UI clearly indicates the "Verifying" state and the final "Verdict".
- [ ] Telemetry correctly captures `ev_verify_success` and `ev_verify_fail`.
