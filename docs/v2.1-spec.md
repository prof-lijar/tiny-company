# Product Specification: TraceWhisper v2.1 \"Debug-First\" Tuning

## 1. Introduction
TraceWhisper v2.1 shifts the product positioning from a "monitoring tool" to an **"IDE for Agent Reasoning."** The core of this transition is the "Debug-First" tuning, which transforms the narrative engine from a storyteller into a forensic analyst.

The goal is to move away from summaries (e.g., "The agent searched for X and then found Y") and toward diagnostic insights (e.g., "The agent is stuck in a reasoning loop: it has called the search tool three times with the same query despite receiving the same result").

## 2. Functional Specifications: \"Debug-First\" Tuning

### 2.1 Forensic Narrative Engine
The LLM prompts used for narrative generation will be overhauled to prioritize the detection of logical fallacies and execution errors.

#### 2.1.1 The Diagnostic Taxonomy
The narrative engine must identify and explicitly label the following "Reasoning Events":
- **[Reasoning Loop]:** When the agent repeats the same thought process or tool call sequence without introducing new information or changing strategy.
- **[Contradiction]:** When the agent's current step contradicts a previous conclusion or a stated constraint in the system prompt.
- **[Strategic Pivot]:** When the agent explicitly recognizes a failure in its current approach and changes its plan.
- **[Tool Hallucination]:** When the agent provides arguments to a tool that are logically impossible or ignores the actual output of a tool in the subsequent step.
- **[Information Gap]:** When the agent attempts a task without the necessary prerequisite information that was available in the context.

#### 2.1.2 Narrative Shift: Summary $\rightarrow$ Audit
- **Old Style (Summary):** "The agent is now checking the weather in London."
- **New Style (Audit):** "The agent is attempting to check the weather in London for the second time. **[Reasoning Loop]**: The previous call already returned 'Sunny', but the agent is ignoring this result."

### 2.2 Debug-First Telemetry & UI
To support the forensic narrative, the UI and data collection must be enhanced.

#### 2.2.1 Visual \"Aha!\" Markers (CLI)
In the `tw live` stream, the system will use visual cues to draw the user's eye to critical failures:
- **Color Coding:** Use `rich` colors (e.g., Bold Red for Loops, Yellow for Pivots).
- **Event Markers:** Prefix narrative segments with the taxonomy labels (e.g., `🔴 [Reasoning Loop]`).
- **Anchor Points:** Ability to "jump" to the first detected loop in a trace.

#### 2.2.2 Forensic Summary Report
At the end of a trace, instead of a general summary, TraceWhisper will generate a **"Logic Audit Report"**:
- **Critical Failures:** A list of all detected Loops and Contradictions.
- **Pivot Analysis:** Where the agent pivoted and why (if successful).
- **Efficiency Score:** A metric based on the ratio of "Productive Steps" vs "Wasted Steps" (loops/hallucinations).

## 3. Technical Requirements
- **Prompt Engineering:** Development of a "Forensic Persona" system prompt for the narrative LLM.
- **State Tracking:** The narrative engine must maintain a window of previous "conclusions" to detect contradictions and loops across a wider context.
- **UI Integration:** Update the `tw live` CLI component to support the new taxonomy markers and color coding.

## 4. Success Metrics

### 4.1 Quantitative Metrics
- **Detection Accuracy:** $\ge 80\%$ agreement between the "Debug-First" labels and manual expert review of traces.
- **Mean Time to Find (MTTF):** Reduce the time a developer spends scanning raw logs to identify a bug by $\ge 50\%$.
- **Loop Detection Rate:** % of traces containing loops that are correctly flagged by the system.

### 4.2 Qualitative Metrics
- **Positioning Alignment:** User feedback explicitly mentioning that the tool feels like a "debugger" or "IDE" rather than a "dashboard."
- **Aha! Moment Frequency:** Increase in users reporting that the tool "caught a bug I didn't see."
