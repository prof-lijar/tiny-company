# Product Specification: v2.2 Interactive Break-points (The Intervention)

**Date:** 2026-05-20
**Status:** Draft / Future Phase
**Priority:** P1 (High)
**Goal:** Transform TraceWhisper from a post-mortem analysis tool into a real-time debugger, allowing developers to "live-patch" an agent's reasoning process.

## 1. Overview
Interactive Break-points allow a developer to pause an agent's execution the moment a reasoning anomaly (e.g., a loop or contradiction) is detected. Instead of letting the agent fail and then analyzing the trace, the developer can intervene, inject a "nudge" (a corrective instruction), and resume execution to see if the agent pivots successfully.

## 2. User Flow

### 2.1 Setting the Break-point
The developer defines the intervention trigger in the SDK during agent initialization:
```python
tw.init(
    breakpoints={
        "on_loop": True, 
        "on_contradiction": True,
        "custom_trigger": lambda trace: "API_ERROR" in trace.last_observation
    }
)
```

### 2.2 The Intervention Event
1. **Detection**: The TraceWhisper SDK monitors the reasoning stream. 
2. **Pause**: As soon as a trigger condition is met, the SDK sends a signal to the agent execution loop to `PAUSE`.
3. **Notification**: The developer's terminal (via `tw live`) flashes a high-priority alert:
   `⚠️ REASONING BREAK-POINT HIT: [Infinite Loop Detected at Step 14]`
4. **State Snapshot**: The CLI displays the current "Reasoning State":
   - The last 3 thoughts.
   - The current goal.
   - The detected anomaly.

### 2.3 The "Nudge" (Injection)
The developer can now interact with the paused agent:
- **Command**: `tw nudge "You are repeating yourself. Instead of searching for X again, try looking for Y in the directory."`
- **Action**: The SDK injects this string directly into the agent's current context/working memory as a "System Intervention."

### 2.4 Resume & Observe
- **Command**: `tw resume`
- **Action**: The agent continues execution.
- **Observation**: The developer observes in real-time if the nudge corrected the path.

## 3. Technical Requirements

### 3.1 SDK Execution Control
- **Pause Mechanism**: The SDK must implement a blocking call or a signal mechanism that can halt the agent's loop without killing the process.
- **Context Injection**: The SDK must have the ability to append text to the agent's prompt history or system prompt dynamically between steps.

### 3.2 CLI Interface
- **Live-Sync**: The `tw live` command must be able to switch from "Streaming Mode" to "Intervention Mode."
- **Input Handling**: A dedicated input prompt for the "Nudge" text.

### 3.3 Integration with Reasoning Engine
- The break-points must use the same detection logic as the `CorrectionEngine` (detecting loops, contradictions, etc.) to ensure consistency.

## 4. Acceptance Criteria
- [ ] **Latency**: The agent must pause within < 1 second of the anomaly detection.
- [ ] **Context Integrity**: The injected nudge must be visible to the agent in the very next step's prompt.
- [ ] **Resumability**: The agent must resume execution from the exact point of the pause without losing previous state.
- [ ] **Custom Triggers**: Developers can successfully trigger a pause using a custom lambda function.

## 5. Success Metrics
- **Correction Velocity**: Reduction in the number of full-run restarts required to fix a reasoning bug.
- **Nudge Success Rate**: % of interventions that lead to a successful goal completion.

## 6. Relation to v2.2 Ecosystem
- **Fix-It**: While 'Fix-It' is for *asynchronous* correction (post-mortem), Break-points are for *synchronous* correction (real-time).
- **Gold Standard**: Successful nudges can be recorded and promoted to the Gold Standard Registry as "Proven Interventions."
