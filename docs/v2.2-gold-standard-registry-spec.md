# Product Specification: v2.2 Gold Standard Registry

## 1. Overview
The Gold Standard Registry is a curated repository of "optimal" reasoning paths for a given set of tasks. It serves as the ground-truth baseline for the **Efficiency Score** calculation and the **CI/CD Reasoning Guardrails**. Instead of relying on stochastic "best-of-N" baselines, the Registry allows teams to explicitly define what a "perfect" reasoning process looks like for their most critical agent workflows.

## 2. The "Gold Standard" Trace
A Gold Standard trace is not just a correct output, but a verified optimal *path*. It contains:
- **The Input**: The exact prompt and context provided to the agent.
- **The Optimal Path**: A sequence of steps (Thought $\rightarrow$ Action $\rightarrow$ Observation) that represents the most efficient way to solve the task.
- **Minimum Step Count**: The number of productive steps in the optimal path.
- **Required Milestones**: Specific steps that *must* occur for the trace to be considered logically sound (e.g., "Must verify API key before calling the endpoint").

## 3. User Flow

### 3.1 Creating a Gold Standard
1. **Execution**: A developer runs a task and achieves a highly efficient, correct result.
2. **Promotion**: The developer marks this specific trace as a "Gold Standard" via the CLI: `tw gold-standard promote <trace_id>`.
3. **Refinement**: The developer edits the trace to remove any minor inefficiencies, creating the "Platonic Ideal" of the path.
4. **Commitment**: The Gold Standard is saved to the registry (local JSON or remote database).

### 3.2 Using the Registry for Evaluation
1. **Comparative Analysis**: When running `tw compare`, the system checks the Registry for a Gold Standard matching the input task.
2. **Efficiency Calculation**: The `EfficiencyCalculator` uses the Registry's `Minimum Step Count` as the denominator for the Efficiency Score.
3. **Guardrail Check**: In CI/CD, `tw guardrail check` compares the new PR's trace against the Gold Standard. If the new trace introduces a loop or deviates significantly from the required milestones, the build fails.

## 4. Technical Specification

### 4.1 Registry Schema (`gold_standards.json`)
```json
{
  "task_id": "user-email-retrieval",
  "version": "1.0",
  "description": "Retrieving user email from CRM and Email API",
  "min_productive_steps": 2,
  "required_milestones": [
    {"step_index": 0, "action": "search_user", "description": "Must first identify the user ID"},
    {"step_index": 1, "action": "get_email", "description": "Must fetch email using the identified ID"}
  ],
  "optimal_trace_ref": "trace_gold_abc123"
}
```

### 4.2 API / CLI Interface
- `tw gold-standard list`: List all defined gold standards.
- `tw gold-standard promote <trace_id>`: Promote a real trace to gold status.
- `tw gold-standard delete <task_id>`: Remove a baseline.
- `tw gold-standard export`: Export the registry for sharing across the team.

## 5. Acceptance Criteria
- [ ] Ability to promote a trace to a Gold Standard via CLI.
- [ ] The `EfficiencyCalculator` correctly pulls `min_productive_steps` from the registry.
- [ ] The `tw guardrail` tool can detect "Milestone Misses" (e.g., the agent skipped a required verification step).
- [ ] Support for versioning Gold Standards (e.g., v1 for GPT-4, v2 for Claude 3.5).

## 6. Integration with v2.2 Ecosystem
- **Efficiency Metric**: Provides the $\text{Minimum Possible Steps}$ variable.
- **A/B Testing**: Allows comparing Prompt A and Prompt B not just against each other, but against the "Perfect" path.
- **CI/CD Guardrails**: Provides the baseline for "Regression Detection."
