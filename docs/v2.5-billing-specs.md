# Technical Specifications: v2.5 Billing System (Platform + Credit Model)

## 1. Overview
The v2.5 billing system transitions Tiny Company from a seat-based model to a **Platform + Credit** model. This system must decouple fixed access costs from variable compute/AI costs to protect margins and provide scalable pricing for the "Autonomous Bridge" features.

## 2. Billing Components

### 2.1 Bridge Platform Subscription (Fixed)
- **Type:** Monthly Recurring Revenue (MRR).
- **Price:** $500 / month / organization.
- **Logic:**
    - Access to v2.5 "Bridge" features is gated by an active subscription.
    - If the subscription is inactive, all v2.5-specific API endpoints must return a `403 Forbidden` or `402 Payment Required` error.
    - The subscription is tied to the `Organization` entity.

### 2.2 Optimization Credits (Variable)
- **Type:** Pre-paid / Consumption-based.
- **Unit:** 1 Credit.
- **Pricing:** $100 per 1,000 Credits.
- **Purchase Flow:** Organizations purchase credit packs (e.g., 1k, 5k, 10k) which are added to their organization's balance.

## 3. Credit Consumption Logic

### 3.1 Consumption Table
The system must deduct credits based on the following activity triggers:

| Activity | Credit Cost | Trigger Point |
| :--- | :--- | :--- |
| **Logic Porter Translation** | 50 cr | Upon successful completion of a translation task |
| **Shadow APO Run** | 200 cr | Upon initiation of a Shadow APO optimization cycle |
| **High-Fidelity Drift Scanning** | 10 cr | Per 100 traces analyzed |
| **Auto-SOP Generation** | 100 cr | Upon generation of a new Standard Operating Procedure |

### 3.2 Deduction Workflow
1. **Pre-Check:** Before executing a v2.5 operation, the system must check if `Organization.credit_balance >= operation_cost`.
2. **Locking:** To prevent race conditions during concurrent requests, a row-level lock on the organization's credit balance is required during deduction.
3. **Deduction:** Subtract the cost from the balance immediately upon the trigger point.
4. **Failure Handling:** If an operation fails due to a system error (not a user error), credits should be refunded or not deducted.

## 4. Metering and Reporting

### 4.1 Usage Tracking
Every credit-consuming event must be logged in a `billing_events` table:
- `event_id`: UUID
- `org_id`: Reference to Organization
- `activity_type`: (e.g., "shadow_apo_run")
- `credits_deducted`: Integer
- `timestamp`: DateTime
- `request_id`: Reference to the specific API request

### 4.2 Reporting Requirements
- **Real-time Balance:** An API endpoint `/billing/balance` to return current credits.
- **Usage Dashboard:** A view for administrators to see:
    - Total credits consumed per month.
    - Breakdown of consumption by activity type.
    - Projected "runway" based on average daily usage.
- **Low Balance Alerts:** Trigger a notification (email/webhook) when credits fall below a threshold (e.g., 200 credits).

## 5. Integration Architecture

### 5.1 Middleware Integration
A billing middleware should be implemented to wrap v2.5 service calls:
`Request` $\rightarrow$ `Auth Middleware` $\rightarrow$ `Subscription Check` $\rightarrow$ `Credit Balance Check` $\rightarrow$ `Service Execution` $\rightarrow$ `Credit Deduction` $\rightarrow$ `Response`

### 5.2 External Payment Gateway (e.g., Stripe)
- **Subscription:** Use Stripe Billing for the $500/mo recurring charge.
- **Credits:** Use Stripe Checkout for one-time credit pack purchases.
- **Webhooks:** Listen for `invoice.paid` to maintain subscription status and `checkout.session.completed` to increment credit balances.

## 6. Edge Cases & Constraints
- **Insufficient Credits:** The system should **not** allow overdrafts unless explicitly enabled by a "Credit Line" flag for specific Enterprise clients.
- **Refunds:** Manual administrative override is required to add/remove credits for customer support reasons.
- **Concurrency:** Ensure atomic updates to the balance to avoid "double-spending" of credits in high-concurrency environments.
