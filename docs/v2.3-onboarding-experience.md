# v2.3 Onboarding Experience: The Path to "Aha!"

## 1. Introduction
The transition from v2.2 (Correction) to v2.3 (Intelligence) is a shift in user psychology. In v2.2, the user was a "Debugger." In v2.3, the user becomes an "Optimizer." 

This document defines the onboarding journey for v2.3, focusing on the "Aha! Moments" that validate the value of the Intelligence Layer.

---

## 2. The v2.3 "Aha! Moments"

We have identified four primary "Aha! Moments" corresponding to the core v2.3 features. The goal of onboarding is to lead the user to these moments as quickly as possible.

### 2.1 The "Collective Intelligence" Moment (Pattern Vault)
*   **The Moment:** The user encounters a complex reasoning failure, and TraceWhisper suggests a fix that was discovered by another project/team in the organization.
*   **The Feeling:** *"I didn't have to spend two hours debugging this because someone else already solved it."*
*   **Trigger:** The first time a "Proven Fix from Organization" is applied and verified successfully.

### 2.2 The "Contextual Clarity" Moment (IDE Integration)
*   **The Moment:** The user clicks a line in a reasoning trace, and the VS Code editor instantly scrolls to the exact line of the system prompt that caused that behavior.
*   **The Feeling:** *"The wall between my code and the agent's mind has disappeared."*
*   **Trigger:** The first time a user uses the "Live-Link" to navigate from Trace $\rightarrow$ Prompt.

### 2.3 The "Invisible ROI" Moment (Cognitive Pruning)
*   **The Moment:** The user runs an Efficiency Audit and sees a specific dollar amount (or token count) they can save per 1,000 runs by removing "Cognitive Bloat."
*   **The Feeling:** *"This tool isn't just a debugger; it's paying for itself."*
*   **Trigger:** The first time the "Tokens Saved" potential is displayed in the Pruning Report.

### 2.4 The "Bulletproof" Moment (CRI & Adversarial Synthesis)
*   **The Moment:** The Adversarial Generator finds a critical edge case that would have crashed the agent in production, and the user fixes it before deployment.
*   **The Feeling:** *"I can actually trust this agent in production now."*
*   **Trigger:** The first time a "Fragility Report" identifies a bug that is then fixed and locked via a `.tw-test` file.

## 3. The Onboarding Journey Map

| Phase | Goal | Key Action | Target "Aha!" |
| :--- | :--- | :--- | :--- |
| **Step 1: Installation** | Zero-friction setup | Install VS Code Extension $\rightarrow$ Connect API Key. | - |
| **Step 2: First Trace** | Visual validation | Open a prompt $\rightarrow$ Run Trace $\rightarrow$ View Side-by-Side. | **Contextual Clarity** |
| **Step 3: The First Fix** | Value realization | Encounter error $\rightarrow$ Apply "Similar Fix" from Vault. | **Collective Intelligence** |
| **Step 4: Optimization** | ROI discovery | Run Efficiency Audit $\rightarrow$ Apply Pruning suggestion. | **Invisible ROI** |
| **Step 5: Hardening** | Trust establishment | Generate Adversarial Tests $\rightarrow$ Create `.tw-test` gate. | **Bulletproof** |

---

## 4. Onboarding Requirements for Design & Engineering

To facilitate this journey, the following "nudges" must be implemented:

1.  **Guided Tour (IDE):** A 3-step walkthrough upon first installation highlighting the Side-by-Side view and the Live-Link.
2.  **The "Vault Welcome":** When a user first opens the Fix-It UI, if no similar fixes exist, show a tooltip: *"As you and your team solve more bugs, the Pattern Vault will start suggesting proven fixes here."*
3.  **Efficiency Nudge:** After the 5th successful trace, trigger a notification: *"Your reasoning looks stable. Want to see if we can reduce your token costs by 20%? [Run Efficiency Audit]"*
4.  **CRI Quick-Start:** Provide a template `.tw-test` file in the onboarding package so users can see `tw verify-all` in action immediately.

## 5. Success Metric for Onboarding
**Time to First "Aha!":** The median time from extension installation to the first successful application of a Vault suggestion or Pruning fix.
**Target:** $< 30$ minutes.
