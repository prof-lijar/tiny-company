# v2.2 Beta Analysis Process

This document defines the operational process for analyzing Beta feedback and the criteria for determining product success for the 'Correction Preview' release.

## 1. Mapping Feedback to the Gold Standard Registry

The Gold Standard Registry is our "source of truth" for optimal reasoning. Beta feedback is the primary engine for expanding this registry.

### The Mapping Workflow:
1. **Capture**: Feedback is logged in `docs/v2.2-beta-feedback-log.md`.
2. **Triaging**: The Product Manager and CTO review the log weekly.
3. **Identification**:
    - **The "Ideal" Path**: If a user provides a manual correction that is significantly more efficient than the AI's attempt $\rightarrow$ **Candidate for Gold Standard Promotion**.
    - **The "Consistent Failure"**: If multiple users report the same failure mode $\rightarrow$ **Gap in Gold Standard**. We must define the "optimal" path for this specific scenario to prevent future regressions.
4. **Promotion**:
    - The optimal trace is extracted from the feedback.
    - The trace is refined to remove noise.
    - The trace is added to the Registry via `tw gold-standard promote`.
5. **Validation**: The new Gold Standard is used to test the 'Fix-It' engine to ensure it can now navigate that path.

---

## 2. Success Signals (The "Value" Indicators)

Beyond raw KPIs (like FSR), we look for these "Success Signals" to confirm the Reasoning IDE is providing genuine value:

1. **The "Aha!" Moment**: A user successfully resolves a high-complexity reasoning failure using 'Fix-It' that they had previously spent >10 minutes trying to fix manually.
2. **Active Registry Contribution**: Users proactively suggesting "This is the perfect way to do this" (indicating they trust the concept of a Gold Standard).
3. **Prompt Iteration Collapse**: A visible shift where the user stops writing long "Please fix this..." prompts and instead relies on the 'Fix-It' button + minimal guidance.
4. **Cross-Task Portability**: A user applying the Reasoning IDE to a completely different set of tasks than the ones they were onboarded with, showing the tool's general utility.
5. **High "Effort Saved" Scores**: Consistent 4s and 5s on the "How much manual effort was saved?" micro-feedback widget.

---

## 3. Beta User Survey: 'Fix-It' Utility Evaluation

This survey is to be sent to Beta users after their second week of usage.

### Survey Title: Evaluating the 'Fix-It' Experience

**Introduction**: *You've been using the Correction Preview for two weeks. We want to know if 'Fix-It' is actually making your life easier or just adding more buttons.*

**Quantitative Questions:**
1. **Accuracy**: On a scale of 1-5, how often does the 'Fix-It' button provide a correct solution on the first try?
2. **Efficiency**: Compared to manual prompt editing, how much time does 'Fix-It' save you per session?
    - [ ] No time saved / It takes longer
    - [ ] Saves a small amount of time (1-5 mins)
    - [ ] Saves a significant amount of time (5-15 mins)
    - [ ] Saves a massive amount of time (15+ mins)
3. **Trust**: How confident are you that the corrected output is logically sound without you having to double-check every step? (1-5)

**Qualitative Questions:**
4. **The Friction Point**: What is the most frustrating part of the current 'Fix-It' loop? (e.g., "It doesn't understand my intent", "The UI is clunky", "It introduces new bugs").
5. **The "Must-Have" Feature**: What is the one thing we could add to the Reasoning IDE that would make it an indispensable part of your workflow?
6. **The Comparison**: Describe a scenario where 'Fix-It' failed completely, but you knew exactly how to fix it. (This helps us build the Gold Standard Registry).

**Closing**: *Thank you for helping us tune the engine!*
